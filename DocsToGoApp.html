Background:
The Docs To Go Application is a useful tool which can be utilized on a daily basis. Furthermore, its capabilities far exceed that of a simple file viewing program. It primarily focuses on documents, spreadsheets, and powerpoint files, enabling the user to view and modify files, as well as generate new ones while on the go. Also, the Docs To Go Application boasts a well designed and aesthetically pleasing formatting window, which uses almost all of the same formatting functionalities as Microsoft Word's toolbar. In addition, its integration with the application, WatchDox, which allows for the highest security possible when sharing sensitive files to other people (see WatchDox App). Combining such a high level of security with the rendering engine of a very capable app results in a optimum formula for efficiency, offering the user the best experience possible from a single application.

My Role:
For the Docs To Go Application, I joined the DTG scrum team (see Docs To Go Team), and worked on smoke testing for the iOS and Android variants. Smoke testing is defined as "preliminary testing to reveal simple failures severe enough to reject a prospective software release". In short, it is a method to check for bugs which may impair the functionalities of the application on its release. Furthermore, the other developers on my team already developed a full smoke test for each of the engines for the application (including Word To Go, Sheet To Go, and Slideshow To Go), which included a HTML file divided into two columns with a condition, then a predicted outcome. After awhile of testing the functionalities of the application, all of the unexpected outcomes are made note of. Next in the process is consulting another group member, and asking for their opinion regarding the severity of the defect. If the bug is confirmed to be legitimate as an acutal issue with the program, then it is filed on a website called Jira. As one might imagine, many of these bug reports reside on Jira. Fortunately, Jira makes it easy to distinguish one report from another, from the many detailed fields which are filled in, regarding what platform the bug affects, its severity, and so forth. Finding and reporting bugs was an interesting process, which not only was a great experience to learn about new applications, but also ensured that the next app's future iterations would be a success. Another type of testing which I conducted is called exploratory testing, which is an antithesis to scripted or smoke testing. Unlike the previous two, exploratory testing does not follow any specific guidelines. Instead, it relies upon the results of previous testing to guide the tester in a certain direction. This more unrestricted mindset has the potential to discover new bugs which may not be visible at first glance. From my experience, it has proved to be successful, as its revolutionary thinking approach covers much more than the obvious.
